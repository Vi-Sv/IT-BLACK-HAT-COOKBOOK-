## 📦 ГЛАВА 2: Цифровой груз (Форматы данных и Валидация)## 📑 Чит №5: Манипуляции с JSON (Древовидная архитектура и обход лимитов)## 🧱 1. Анатомия системы (Суть процесса)
JSON (JavaScript Object Notation) — это текстовый, открытый стандарт обмена данными. Он полностью вытеснил старые форматы благодаря двум свойствам: его одинаково легко читает человек глазами и парсит (разбирает на объекты) любой язык программирования.
Физически JSON — это текстовая строка, построенная на строгом сочетании всего двух архитектурных конструкций:

   1. Объект (Словарь / Ассоциативный массив): Заключается в фигурные скобки {}. Хранит данные в виде пар «Ключ»: Значение. Ключ — это всегда текстовая строка в двойных кавычках.
   2. Массив (Список): Заключается в квадратные скобки []. Хранит упорядоченный список значений любого типа, разделенных запятыми.

JSON поддерживает всего 6 примитивных типов данных: строка, число, логический тип (true/false), массив, объект и пустота (null).

Запомнить навсегда: Текстовая строка JSON — это цифровая иерархическая матрешка. Фигурные скобки {} — это коробка с именными ячейками. Квадратные скобки [] — это длинный конвейерный лоток, где коробки или детали лежат по порядку (по индексам от 0 до N). JSON не умеет хранить формулы или циклы, это строго снимок застывших данных.

## 🪤 3. Канализация системных ошибок и капканов (Граничные условия)

* Капкан №1: Синтаксический брак (Запятая-убийца / Trailing Comma).
Самый частый ручной косяк при ручном формировании JSON. По спецификации стандарта, после последнего элемента в объекте или массиве запятая ставить запрещено.
* Ошибка: Если твой генератор выдаст строку [1, 2, 3,] или {"id": 1,}, встроенный парсер сервера моментально сломает зубы, прервет обработку и выплюнет ошибку 400 Bad Request (Синтаксический брак). Строка не распарсится в объект.
* Капкан №2: Оверклокинг RAM при глубокой вложенности (Матрёшка смерти).
JSON позволяет вкладывать объекты друг в друга бесконечно. Но когда сервер принимает этот текст, он обязан развернуть его в оперативной памяти (RAM) в виде дерева объектов.
* Ошибка: Если злоумышленник или кривой скрипт пришлет JSON с глубиной вложенности в 1000 уровней ({"a":{"b":{"c":...}}}), серверный парсер сожрет всю RAM на построение связей, вызовет переполнение стека (Stack Overflow) и намертво уронит приложение на проде.
   * Решение Аналитика: В системных требованиях (ТЗ) к API аналитик жестко прописывает лимиты: максимальный размер пакета в байтах и ограничение на уровень вложенности JSON (обычно не более 5-10 уровней).
* Капкан №3: Потеря точности больших чисел (BigInt капкан).
Стандарт JSON не разделяет числа на целые и дробные — для него всё это тип Number (64-битное число с плавающей точкой).
* Ошибка: Если ты передаешь в JSON точный ID транзакции или номер вагона длиннее 15 знаков (например, 9007199254740993), при парсинге в JavaScript или Python последние цифры округлятся в ноль из-за физических ограничений точности плавающей точки. Данные испорчены.
   * Решение: Большие ID, номера документов, телефоны и номера вагонов в JSON аналитик обязан проектировать строго как тип Строка (в кавычках ""), а не как числа.

## 🎯 4. Срез для собеседования в BigTech (Что спросит Лид)

* Вопрос: Что такое Сериализация и Десериализация JSON, и на каком этапе расходуется больше всего ресурсов процессора?
* Ответ бьющий в ядро:
* Сериализация — это перевод живого объекта из оперативной памяти кода в текстовую строку JSON для отправки по трубе.
   * Десериализация (Парсинг) — обратный процесс: превращение текстовой строки JSON в живой объект в RAM сервера.
   Процессор сильнее всего нагружается на этапе Десериализации, потому что коду нужно посимвольно прочитать весь текст, проверить его на синтаксические капканы (запятые, кавычки), выделить память под каждый ключ и построить дерево связей. На высоконагруженных (Highload) системах Бигтеха кривой парсинг тяжелых JSON — это главный убийца производительности CPU.
