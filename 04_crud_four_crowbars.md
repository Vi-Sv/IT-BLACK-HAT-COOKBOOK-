##  ГЛАВА 2: Цифровой груз (Форматы данных и Валидация)
## Чит №4: Четыре лома CRUD (Анатомия перманентного хранения)
##  1. Анатомия системы (Суть процесса)
Абсолютно любое действие с данными в ИТ-системах (в приложении, на сайте, в СУБД) сводится к четырём базовым операциям — концепции CRUD: Create (Создать), Read (Прочитать), Update (Обновить), Delete (Удалить).
На транспортном уровне HTTP эти операции выполняются с помощью четырёх основных методов (директив), которые зашиваются в стартовую строку запроса. Это четыре главных лома, которыми аналитик заставляет сервер ворочать строки в базах данных.

* GET (Read — Прочитать): Запрос на выдачу ресурса. Он безопасен и пассивен. GET-запрос не имеет права ничего менять на сервере. Сколько бы раз ты его ни вызывал, база данных должна оставаться неизменной. У него, как правило, нет тела (Body) — параметры поиска зашиваются прямо в строку адреса.
* POST (Create — Создать): Запрос на создание новой сущности. Он несёт в теле (Body) тяжелый груз — данные нового объекта (например, параметры нового шлейфа или акта). Каждый вызов POST генерирует новую уникальную строку в базе данных.
* PUT / PATCH (Update — Обновить): Запросы на изменение данных.
* PUT — заменяет объект целиком (перезаписывает все поля строки новыми значениями).
   * PATCH — точечный ремонт (меняет только одно конкретное указанное поле, не трогая остальные).
* DELETE (Delete — Удалить): Запрос на уничтожение ресурса или перевод его в статус неактивного (архивного).

Запомнить навсегда:
GET — это заглянуть в Журнал сварочных работ (просмотр).
POST — это вклеить в Журнал новый чистый лист для следующего стыка.
PUT — это вырвать старый лист и вставить вместо него полностью переписанный новый.
PATCH — это аккуратно зачеркнуть одну неверную дату на листе и написать сверху правильную.
DELETE — это залить лист чернилами или выбросить в шредер.

##  2. Логическая цепь: Идемпотентность (Главный инженерный замок)
Центральное понятие в методах CRUD, на котором валятся 95% джунов на собеседованиях в Бигтех — это Идемпотентность.

Идемпотентность — это свойство метода выдавать один и тот же результат при многократном повторении запроса.

Давай пропустим методы через эту математическую проверку:

   1. GET — Идемпотентен: Если ты запросишь карточку заказа №408 один раз или 10 000 раз подряд, сервер будет возвращать тебе одну и ту же карточку (если её не изменил кто-то другой). База данных стабильна.
   2. PUT / PATCH / DELETE — Идемпотентны: Если ты отправишь запрос «Запиши в строку №408 статус "Перебор"», то сколько бы раз ты потом этот запрос ни дублировал, статус строки останется "Перебор". Результат системы зафиксирован. Если ты отправишь «Удали строку №408», первый запрос её удалит, а последующие будут возвращать 404 Not Found, но физический результат на сервере не изменится — строки как не было, так и нет.
   3. POST — НЕ ИДЕМПОТЕНТЕН: Если из-за сбоя сети твой клиент отправит запрос «Создай заказ на 3100 тонн масла» 5 раз подряд, ленивый сервер создаст в базе данных 5 абсолютно разных строк с разными ID и спишет деньги 5 раз. POST каждый раз плодит новые сущности.

##  3. Канализация системных ошибок и капканов (Граничные условия)

* Капкан №1: Двойной клик пользователя (Ловушка дублирования POST).
Пользователь нажимает кнопку «Оплатить», интернет моргнул на этапе отправки ответа, кнопка не заблокировалась, и пользователь в панике жмет её еще раз. Летит второй POST.
* Чем грозит: В базе создается дубликат финансовой транзакции. Списание х2.
   * Решение Аналитика (Ключ идемпотентности): Системный аналитик заставляет фронтенд генерировать уникальный случайный хэш для сессии — Идемпотентный ключ (Idempotency Key) — и совать его в заголовок POST-запроса. Если сервер видит, что прилетел POST с ключом, который он уже обработал 5 секунд назад, он не лезет в базу создавать новую строку, а просто дублирует клиенту старый успешный ответ. Контрабанда дублей заблокирована.
* Капкан №2: Превышение лимитов строки URL в GET.
Так как у GET-запроса нет тела, неопытные разработчики пытаются запихать тяжелые массивы данных прямо в строку адреса (Query-параметры): ?id=1&id=2....
* Ошибка: Серверные веб-серверы (например, Nginx) имеют жесткое физическое ограничение на длину строки URL (обычно 2–8 Кб). Как только лимит превышен, сервер моментально обрубает соединение и выплевывает код 414 URI Too Long. Данные не долетели.
   * Решение: Если нужно передать массив фильтров для чтения, аналитик меняет метод на специальный POST-запрос для чтения (так делают в ElasticSearch) или пакует параметры в тело.
* Капкан №3: Затирание данных при PUT (Отрезание хвостов).
Если ты используешь PUT для обновления карточки сотрудника, ты обязан прислать в теле вообще все поля, даже те, что не менялись. Если ты пришлешь только {"salary": 150000}, а поля name и position забудешь указать, ленивый PUT послушно запишет зарплату, а вместо имени и должности запишет NULL. Человек стерт из системы.
* Решение: Для частичных апдейтов всегда проектировать строго метод PATCH.

##  4. Срез для собеседования в BigTech (Что спросит Лид)

* Вопрос: В чем разница между Безопасными (Safe) и Идемпотентными (Idempotent) методами HTTP?
* Ответ бьющий в ядро: Безопасные методы — это те, которые вообще не изменяют состояние ресурсов на сервере (чистое чтение). Это метод GET. Идемпотентные методы — это те, которые могут изменять данные, но повторные вызовы этих запросов не меняют результат системы дальше первого удара (PUT, PATCH, DELETE). Таким образом, любой безопасный метод автоматически является идемпотентным, но не любой идемпотентный метод безопасен для данных.
